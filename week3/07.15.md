# be-DailyJob 

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: WAS 라우팅 테이블 개발 완료, 데이터 모델링 학습(타입, 컬럼별 특징 및 숙지 사항 정리), 자바 함수형 인터페이스/람다식/메서드 참조(`::`) 동작 원리 파악 및 코드 적용
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: WAS 라우터 전체 기능 연동 완료

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: WAS 라우터 개발 시, 한 번만 사용하는 라우팅 로직을 익명 클래스가 아닌 람다식과 메서드 참조를 적극 활용하여 구현하기로 결정함.
- 그렇게 판단한 이유: 라우터 특성상 인터페이스 다형성을 활용해 구현체를 갈아끼우는 코드가 많은데, 매번 `new Machine() {...}` 형태로 익명 클래스를 선언하면 코드가 길어지고 핵심 로직(라우팅 목적지)이 가려지기 때문.
- 고려했지만 선택하지 않은 대안: 전통적인 `implements`를 활용해 별도의 구현체 클래스를 파일로 분리하는 방법. 한 번만 쓰이는 로직을 위해 클래스를 늘리는 것은 오버엔지니어링이라 판단함.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 불필요한 보일러플레이트 코드(메서드 선언부 등)가 제거되어, 라우팅 테이블의 가독성이 비약적으로 상승함.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: WAS 라우터 개발 중 익명 클래스와 람다식을 혼용하는 과정에서 자바 문법에 막혀 코드 작성이 지체됨.
- 처음 추측한 원인: 람다식과 메서드 참조를 단순히 코드를 줄여주는 '축약어' 정도로만 얕게 이해하고 있어 매개변수 처리가 헷갈렸다고 생각함.
- 원인을 확인하기 위해 시도한 방법: 코드를 역추적하여 명시적 오버라이드 → 람다식 축약 → 메서드 참조로 이어지는 코드 변환 과정을 직접 작성하며 테스트함.
- 실제 원인: 인터페이스(다형성)에서 시작해 메서드 참조까지 이어지는 자바 컴파일러의 타입 추론 메커니즘을 정확히 숙지하지 못하고 감으로 코딩하고 있었음.
- 해결 방법: 파라미터를 그대로 넘기는 로직(`area -> System.out.println(area)`)은 변수 지정 없이 `System.out::println`으로 치환 가능함을 원리적으로 이해하고 라우터 코드에 적용함.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: IDE의 자동 완성에 의존하기 전에, 내가 구현하려는 인터페이스가 단일 추상 메서드를 가진 '함수형 인터페이스' 조건에 맞는지 명확히 확인할 것.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 문법 에러가 났을 때 구글링하여 돌아가는 코드만 복붙하지 않고, 다형성부터 메서드 참조까지의 내부 발전 과정을 논리적으로 뜯어본 것.
- 그 접근이 도움이 된 이유: "왜 굳이 변수를 지정해 줘야 하지?"라는 의문을 스스로 던지고 답을 찾았기에, 이후 라우팅 테이블 작성 시 확신을 가지고 코드를 작성할 수 있었음.
- 다음에도 반복하고 싶은 점: 구현 중 막히는 개념이 나오면, 잠시 멈추더라도 근본적인 원리와 언어의 설계 의도를 먼저 파악하고 넘어가는 태도.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 자바 기본 문법 숙련도 부족으로 인해, 실제 WAS 라우터의 비즈니스 로직 개발에 집중해야 할 시간을 문법 이해에 꽤 많이 소모한 점.
- 그렇게 된 원인: 평소 코드를 짤 때 기존에 작성된 코드를 관성적으로 참고하거나, 문법의 형태만 외워서 사용했기 때문.
- 다음에는 다르게 해볼 구체적인 행동: 자주 사용하는 자바 8+ 기능들(Stream, Optional, Lambda 등)은 실무에서 문법으로 더 이상 고민하지 않도록, 자투리 시간을 활용해 딥다이브 형태로 문서화를 진행할 것.
