### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 람다식과 스트림(Stream)의 동작 원리 파악 및 적용 기준 확립, Optional 지연 평가(`orElseThrow`) 원리 학습
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 홈페이지의 기능 구현 부분 (게시글, 댓글, 좋아요 등)

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 데이터 조회 및 가공(DTO 변환 등)에는 스트림을 적극 도입하되, 트랜잭션이 발생하거나 복잡한 try-catch가 얽힌 DB Insert/Update 로직에는 전통적인 for문을 유지하기로 함.
- 그렇게 판단한 이유: 스트림 파이프라인 내에서 예외를 처리하거나 상태를 변경하는 것은 함수형 프로그래밍의 순수성 원칙을 해치고, 디버깅 및 트랜잭션 관리를 어렵게 만들기 때문.
- 고려했지만 선택하지 않은 대안: 모든 반복문을 스트림으로 일괄 전환하는 것. 코드 길이는 짧아지겠지만 제어가 어려워져 시스템 안정성을 해칠 수 있다고 판단함.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: API DTO 변환 로직의 가독성은 극대화하면서도, 핵심 비즈니스 로직의 트랜잭션 안정성은 지킬 수 있게 됨.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 기존 for-if문에 익숙한 상태에서 람다식(`->`)과 메서드 참조(`::`), 스트림 파이프라인 내부의 변수(예: `s`, `p`) 정체를 파악하지 못해 코드 흐름을 읽기 어려웠음.
- 처음 추측한 원인: 람다식을 기존 객체지향과 동떨어진 완전히 새로운 문법이나 단순한 축약형으로만 접근함.
- 원인을 확인하기 위해 시도한 방법: 자바 컴파일러가 스트림 연산 시 어떻게 제네릭 타입을 역추적하여 파라미터를 식별하는지 다형성과 익명 클래스 구조부터 다시 뜯어봄.
- 실제 원인: 인터페이스 변수 = `new 구현체()` 형태에서 파생된 '1회용 익명 클래스'가 람다의 본질임을 인지하지 못함.
- 해결 방법: 다형성 → 익명 클래스 → 람다식 → 메서드 참조로 이어지는 자바의 함수형 패러다임 진화 과정을 구조적으로 이해함. 파라미터를 그대로 토스하는 경우 변수명마저 생략 가능함(`.map(PaymentDto::new)`)을 깨달음.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 축약된 코드에 압도되지 말고, 컴파일러가 어떤 인터페이스(함수형 인터페이스)를 기반으로 타입을 추론하고 있는지 원형을 먼저 복원해볼 것.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: `Optional.orElseThrow(() -> new IllegalArgumentException())` 코드를 작성할 때, 단순히 에러를 던지는 용도로만 외우지 않고 '왜 람다식을 넘기는가?'에 대해 파고든 것.
- 그 접근이 도움이 된 이유: 람다식(Supplier)을 넘김으로써 객체가 null일 때만 예외 객체를 생성하도록 유도하는 '지연 평가(Lazy Evaluation)'의 원리를 깨닫고, 람다의 진짜 효용성을 이해할 수 있었음.
- 다음에도 반복하고 싶은 점: 단순히 타이핑 타이밍을 줄이기 위한 문법적 설탕(Syntactic Sugar)으로 치부하지 않고, 그 이면에 있는 메모리나 성능적 이점까지 연결해서 사고하는 습관.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 람다와 스트림의 원리를 모른 채 축약형부터 보려다 보니, 코드 리딩에 필요 이상의 시간을 소모함.
- 그렇게 된 원인: 람다식에 대해 이해하지 않고 넘겼던 경우가 많아, 자바의 핵심인 다형성/객체지향과 연결 지어 생각하지 못했기 때문.
- 다음에는 다르게 해볼 구체적인 행동: 새로운 문법이나 기술을 마주할 때, 어떻게 쓰는가보다 기존의 어떤 문제를 해결하기 위해 등장했는가와 그 근간이 되는 원리를 먼저 탐구하는 방식을 취할 것.
