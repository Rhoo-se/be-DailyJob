# be-DailyJob 

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  a. HTML 응답을 정적 -> 동적 HTML로 변경하였다.
  b. HttpRequest에서 담당하던 HTTP 요청별 분기처리를 Router 클래스 및 라우터 맵을 통해 책임 분리 하였다.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  b. 라우터를 도입하며, 라우터 테이블을 만들어야했고, 분기처리 이후 이어지는 HTTP 응답의 책임을 기존 HttpResponse에서 
  RouteHanlder 인터페이스의 handle()로 위임하는 부분을 완료하지 못하였음

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  b. Map형태의 라우터 테이블을 만들고, 이 Map의 value인자로 함수형 인터페이스를 사용한 것이 좋은 판단이었다고 생각한다.
  
- 그렇게 판단한 이유:
  b. 다음 미션으로 넘어갈수록 좋아요, 게시판, 댓글 등 여러 클래스들의 기능들이 늘어나고, 이것들은 로그인 여부, 작성자 여부 등에 따라 반환해야하는 응답 값이 매우 다양해지게 된다.
  이에 기존 if문으로 분기처리통해 정적/동적 파일을 반환하는 방법은 기능이 추가될수록 유지보수를 어렵게 하는 방법이라 생각했다.
  그렇지만 모든 응답을 담을 수 있는 Spriing의 ResponseEntity같은 객체를 직접 만들기에는 아직 부담이 크다 생각하였고, 이에 각 클래스들이 직접 자신의 응답을 갖고 반환할 수 있도록
  RouterHandler를 상속하여 handle()이라는 추상메소드가 이를 담당하도록 하였다.
  
- 고려했지만 선택하지 않은 대안:
  b. ResponseEntity와 같은 HTTP의 응답을 담을 수 있는 객체를 만들 수 있었지만, 현재 프로젝트 규모대비 불필요한 개발 지점이 너무 많다 생각하여 채택하지 않았다.
  
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  b. 요청 주소가 늘어날수록, 간단하게 map에 추가하면 되므로 코드의 가독성이 좋아졌다.
  또한 각자의 클래스(도메인)이 자신의 HTTP응답까지 담당하여 RequestHandler의 역할이 명확해지게 되었다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  a. 코덱스 리뷰에 의해 동적으로 String 형식의 HTML을 생성하면 XSS 공격과 같은 취약점이 생길수 있다는 사실을 알게 되었다.
  
- 처음 추측한 원인:
  a. 사용자가 로그인/비밀번호/회원정보를 입력하는 폼에, 직접 스크립트를 입력할 수 있었기 때문이라고 생각했다

- 원인을 확인하기 위해 시도한 방법:
  a. 사용자가 실제로 입력하는 값을 그대로 사용하는 부분이 있는지 확인하였고, user.getName()과같은 메서드를 사용하는 것을 찾게 되었다. 
  
- 실제 원인:
  a. 생각한 원인이 맞았다.

- 해결 방법:
  a. escapeHtml()이라는 메서드를 만들어 위험한 특수문자들(&, <, >, \, ')을 HTML엔티티로 치환하였다.
  
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  a. XSS, SQLInjection과 같이 사용자의 입력을 직접 갖다쓰고 있는 코드는 없는지, 이를 어떻게 대비하고 있는지 다시 한 번 확인하게 되었다.
