### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 프로젝트 전체 스코프를 스트림(메인/서브/애디셔널/퍼스널/딜리티드)으로 분류하고 우선순위 확정
  - 결제/정산/포인트에 대한 방향 확정: 실제 PG는 제외(딜리티드), 가상 포인트로 결정
  - 하향경매 사용자 이해도 문제 인식 및 캐치프레이즈 방향 도출
  - MVP → 고도화 → 부하테스트 → 모니터링 순차 진행 계획 수립
  
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 실제 코드 착수
  - 와이어프레임 완성 및 API 명세서

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 결제/정산/충전/출금을 실제로 구현하지 않고, 폐쇄형 가상 포인트 경제(가입 시 일괄 지급 + 거래내역 원장)로 대체하기로 결정.
    
- 그렇게 판단한 이유:
  - 실제 결제/정산은 PG 연동이 필요해 4주 범위를 벗어남
  - 충전/출금을 가짜로 구현하면 실체 없는 껍데기가 됨
  - 원장·트랜잭션·동시성 등을 보여주기 위해 가상 포인트를 활용하면 도메인 및 기술적 깊이를 학습할 수 있음
    
- 고려했지만 선택하지 않은 대안:
  - 실제 PG(토스페이먼츠 등) 연동 → 범위 초과로 제외(결제, 에스크로 등)
 
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 실제 결제 로직이 없어 외부 의존성이 줄고, 테스트/유지보수 부담이 낮아짐 

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - "결제/정산/수수료를 하려면 포인트·충전·출금이 다 필요하다"는 결론에 도달해 스코프가 계속 커짐. 동시에 "돈을 관리 안 하면 판매자가 정산을 못 받는다"는 모순에 부딪힘.
  
- 처음 추측한 원인:
  - 포인트/충전/출금 기능을 다 만들어야 거래가 완결된다고 추측
  
- 원인을 확인하기 위해 시도한 방법:
  - "충전/출금이 필요한 진짜 이유가 무엇인가"를 되짚음 → 그것이 "실제 돈과 플랫폼을 잇는 다리"임을 확인. PG가 없으면 그 다리가 이어줄 대상이 없다는 점을 확인
  
- 실제 원인:
  - "돈의 이동"과 "돈의 기록"을 혼동한 것. 판매자 정산 표시는 잔액 관리(포인트)가 아니라 거래내역 집계(SUM)로 해결되는 문제였음. 사기 방지(에스크로)만이 진짜로 돈을 쥐어야 하는데, 그건 PG 영역(범위 밖).

- 해결 방법:
  - 실제 돈 이동은 당사자 간(플랫폼 밖)으로, 플랫폼은 상태·기록만 담당
  - 정산 표시는 낙찰가 집계 + 수수료 계산으로 해결(포인트 불필요)
  - 별개로, 데모 완결성을 위해 폐쇄형 가상 포인트 경제를 선택적으로 도입
  - 판매자 미배송(사기)은 신고→분쟁→제재로 사후 대응, 근본책(에스크로)은 로드맵으로 명시
    
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 이 기능이 필요하다고 느끼는 진짜 이유가 뭔가를 먼저 확인하기. 전제가 성립 안 하면 기능 전체가 불필요할 수 있음.
