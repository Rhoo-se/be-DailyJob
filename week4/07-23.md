### 2. 오늘 가장 의미있었던 기술적 판단

**(1) 인증 방식: Session 채택 (vs JWT)**

- 내가 내린 판단: MVP 단계에서는 Session 방식을 채택한다.
- 그렇게 판단한 이유: 단일 서버 환경이라 구현이 빠르고, JWT의 핵심 장점인 "분산 서버 환경에서의 간단한 서명 검증"이 애초에 필요가 없어지는 상황이라고 판단했다.
- 고려했지만 선택하지 않은 대안: JWT — 분산 환경 이점이 사라지는 반면, 오히려 로그아웃/만료 관리가 어려워진다는 단점(스테이트리스라 서버 측에서 즉시 무효화하기 어려움)이 부각되어 기각.
- 이 판단이 품질/유지보수에 미친 영향: 로그아웃·만료 처리는 단순해지지만, 세션 저장 방법과 향후 스케일아웃 시 세션 공유 방안을 별도로 대비해야 하는 부채가 생김.

**(2) 영속성 기술: Spring JPA 채택 (vs JdbcTemplate / Spring Data JDBC)**

- 내가 내린 판단: 영속성 계층은 Spring JPA로 구현한다.
- 그렇게 판단한 이유: 자바(객체지향)와 관계형 DB(테이블 중심) 사이에는 연관관계·상속·방향성 등에서 근본적인 간극이 있는데, JPA는 이 간극을 메워 SQL이 아니라 객체를 다루듯 코드를 짤 수 있게 해준다. 복잡한 객체 관계 표현, 변경 감지와 작업단위 관리로 생산성이 높다는 점을 중요하게 봤다.
- 고려했지만 선택하지 않은 대안:
   - Spring Data JDBC — 지연 로딩·변경 감지가 없어 동작이 단순하고 예측 가능하며, CRUDRepository로 기본 쿼리를 제공하고 애그리거트 경계를 강제하는 DDD 기반 구조라는 장점이 있었지만, JPA만큼의 객체 관계 표현력과 생산성은 포기해야 해서 이번엔 기각.
   - JdbcTemplate — WAS 시절부터 익숙한 방식이라 비교는 해봤으나, 보일러플레이트가 많고 생산성이 떨어져 기각.
- 이 판단이 품질/유지보수에 미친 영향: 생산성은 높아지지만 실행되는 SQL을 놓치기 쉽고(지연 로딩으로 인한 N+1 문제), 영속성 컨텍스트의 숨은 비용(변경 감지, 동일성 보장, 쓰기 지연, 1차 캐시)을 관리해야 하며, 복잡한 쿼리 작성 시 어려움이 예상되어 이에 대한 대응이 필요함.
